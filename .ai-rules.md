# SafeZone (Application) AI Rules

This directory contains the source code and business logic for the SafeChord application.

## 1. Directory Structure
Understanding the internal layout of `SafeZone/` is crucial for the correct placement of code and logic.

```text
SafeZone/
├── data/           # Static data files and smoke test scenarios.
├── docker-compose/ # Local development & testing container orchestrations.
├── scripts/        # CI/CD and utility shell scripts (build, test, deploy).
├── services/       # Core microservices (e.g., analytics-api, dashboard, data-ingestor).
├── toolkit/        # Internal tools and domain-specific CLIs (e.g., szcli).
└── utils/          # ⚠️ Git Submodule: Shared utilities (Pydantic models, loggers, DB clients).
```
**Important:** The `utils/` directory is a **Git Submodule**. Any modifications to shared models or loggers within `utils/` must be committed and pushed according to submodule workflows before updating the reference in `SafeZone/`.

## 2. Core Standards
- **Container-First**: Do NOT run runtimes (Python, Go, Node) directly on the host. Use the provided `Makefile` commands or Docker.
- **Python Development**: Use **Pydantic** for all data validation and modeling.
- **Observability**: Structured JSON logs and Trace ID propagation are mandatory for all services.

## 3. Testing Guardrails & TDD Workflow
- **"Test is the Law"**: New logic MUST be verifiable.
- **TDD Requirement (Services Layer)**: When implementing or modifying business logic in `SafeZone/services/`, you **MUST** follow a Test-Driven Development approach:
  1. **Write/Update Tests First**: Before writing the actual implementation, locate the corresponding unit/integration test file and write the test cases that cover the new requirements.
  2. **Verify Failure (Red)**: Run the tests using the appropriate `make test-<service>` command to confirm the new tests fail.
  3. **Implement (Green)**: Write the actual service code to satisfy the test requirements.
  4. **Verify Success**: Run the tests again to confirm they pass.
- **Smoke Tests**: All new features or major logic changes must include corresponding E2E verification logic in `SafeZone/data/smoke-test/`. *(Note: You do not need to run the smoke tests locally; the CI pipeline will execute them).*

## 4. Automation, CI/CD, & Key Commands (Router)
The build, test, and deployment automation for this repository is strictly defined in the `Makefile` and GitHub Actions workflows. **Do NOT invent your own commands** (e.g., running raw `pytest`, `docker build`, or `go test` directly) unless explicitly requested.

🛑 **Mandatory Triggers:**
- **For local building, testing, and dev environments:** You **MUST** read `SafeZone/Makefile` to understand the available targets (e.g., `make build-<service>`, `make test-<service>`, `make dev-up`). The `Makefile` handles the complex containerized build/test logic via scripts.
- **For CI/CD pipeline understanding:** If you are unsure about the official CI process, check the workflows in `SafeZone/.github/workflows/`.
